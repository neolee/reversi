# The REAL README 

*by Human Author of This Project*

这是这个项目中唯一一个完全由人类（我，[neolee](https://github.com/neolee)）编写的文件，其他代码和文档 99% 都由 AI 根据我的指令生成。

## 项目动机

这个项目的原始动机来自我在创作《算法》课程及教材时的想法，我希望通过一个博弈类游戏来展示经典 AI 算法（如 Minimax 和 Alpha-Beta 剪枝）的实际应用。Reversi（又称 Othello）作为一个规则简单但有相当策略深度的游戏，非常适合用来教学和演示这些算法，实际上 Reversi 也是多年以前我在大学时做过的一个实例，颇有感情。

同时，我也希望利用一个有相当复杂度的项目来考验当下最先进的 AI 编程工具，看看它们在实际项目开发中的表现如何。因此，这个项目既是一个教学工具，也是一个 AI 编程能力的实验场。

## 工具

- 集成开发环境：Visual Studio Code
  - 目前仍是能力最综合也最容易上手的集成开发环境；
- Coding Agent：GitHub Copilot
  - 目前各类 coding agent 的能力已大致拉平，由于 Copilot 可以方便测试多种不同的主流大语言模型，成为我的首选；
- 大语言模型：GPT-5.1-Codex，Gemini 3
  - 这是我认为目前最强的编程领域大模型，在本项目的过程中，我交替使用这两种模型，并比较它们的表现。

经过这个项目的实践，我的一些初步发现：
- GitHub Copilot 的能力已经相当完善，和 Visual Studio Code 的集成非常好，完全不弱于曾经领先的 Cursor、Claude Code、Codex 等编程 agent；
- GPT-5.1-Codex 和 Gemini 3 在编程任务上不相上下，但综合表现 GPT-5.1-Codex 略胜一筹；
- 今后我会倾向于在不同任务上选择不同的模型：
  - 讨论需求与设计思路时优先使用 Gemini 3，其次是 GPT-5.1-Codex；
  - 执行比较复杂的编码任务时，优先使用 GPT-5.1-Codex，其次是 Gemini 3；
  - 简单的重构可以使用 Grok Code Fast 1 这样的免费替代。

下次有机会我还会试试国内的 DeepSeek、千问、Kimi K2 等模型的能力。

## 项目历程

### 确定目标

> 目前使用 AI 辅助编程的关键在于需求和架构设计的明确及细致程度，一种最佳实践是与 AI 先就产品目标和关键设计进行讨论，生成尽可能详尽的软件目标规格说明文档，然后再让 AI 根据这个文档生成代码。

第一步，我先编写了一份非常简单的概要，即最初版本的 [README.md](https://github.com/neolee/reversi/blob/637d5db8f2adaf101c0c43b5881304e355a4c6a1/README.md)，里面包含了我仔细挑选的核心关注，包括：
- 项目核心目标包含教学与娱乐两方面
- GUI 和引擎的解耦设计
- 既支持 Python 编写的内置引擎，也支持包装引入第三方引擎
- 选用合适的跨平台 GUI 框架
- 使用 Python 主流版本和流行工具

第二步，经过多轮讨论，Gemini 3 帮助我丰富完善了这份 README，主要包括两个新的功能点：棋局保存/加载/回放以及引擎间 headless 对战，并初步确定 GUI 候选框架，这就是 README 的[更新版本](https://github.com/neolee/reversi/blob/b39b07e83e8a110c4ba044dd7eac1d3a7a8a39b0/README.md)。

第三步，在此基础上要求 AI 生成一个供 coding agent 使用的项目记录文档即 [AGENTS.md](https://github.com/neolee/reversi/blob/60970ef5937494328ac8b2e7b3499c06d81c6b34/AGENTS.md)，这个文档记录项目目标、关键设计和技术决策、项目现状、未来规划以及其他有用的笔记，以便 coding agent 能随时快速了解关键上下文。

同时我也要求 coding agent 在生成代码前阅读这个文档，以确保代码生成符合项目目标和设计原则，并在关键里程碑后更新这个文档。这实质上是帮助 AI 建立一个简明的“关键记忆体”，效果比完全依赖 coding agent（目前极不完善）的自身记忆能力要好得多。

> 有趣的是，Gemini 3 一开始没理解我的意图，它以为我让它为项目设计一组 AI agents（这非常奇怪，因为理解项目目标之后应该知道这个项目并不会基于 AI agents 来实现），我重新解释后它才明白过来并生成了正确的内容。

> 严格来说，目前的 AGENTS.md 文档还不十分符合我的预期，我希望它能记录 coding agent 在项目开发过程中做出的所有关键决策和思考过程，去掉不必要的细节和已经过时的信息，也许以后可以通过一套完善的提示词来让 coding agent 自动维护这个文档。

### 项目里程碑

在确定了项目目标和设计后，我和我的 AI 助手进一步讨论项目实现的路径，AI 倾向于先实现引擎部分，而我最终选择了先实现 GUI 部分，原因是：
1. 我完全不了解 Python 的 GUI 框架，需要探索和选型，且由于 AI 无法自行与 GUI 应用交互，开发过程高度依赖我作为测试员和信息中介，因此这是整个项目最有风险的部分，需要优先解决；
2. GUI 部分的实现可以提供一个丰富的可视化环境，使我们每前进一小步都能清晰看到效果，这对于 AI 协同编程来说非常有帮助；
3. GUI 的实现过程也有助于我们进一步细化产品功能和交互细节的设计，从而为后续的引擎及协议实现提供更具体的需求，也就是把 GUI 部分作为可迭代原型来用。

> 从这一点可以看出，在 AI 可以完全独立完成复杂产品开发前，AI 适应人类开发者的工作方式是非常重要的，我的偏好并不一定普适，但如果人类开发者不了解自己需要什么的话，合作的效率会非常不可期。

在接下去的开发过程中，我和 AI 助手一起经历了多个里程碑，每个里程碑基本上都遵循了类似的流程：
1. 我给出里程碑的需求以及相关约束，如果需要我会先和 AI 讨论特定要点，但最后决定权在我；
2. Coding agent 根据需求生成代码，我会大致确认代码修改点，如果发现路子明显不对就要求它改（这种情况不多但有）；有时候它生成的代码我看不懂，也会要求它解释（这在 GUI 部分很常见，因为是我之前没碰过的领域）；
3. 实际运行程序进行测试验证，这需要我设计合理的测试方法，并精确地向 AI 反馈发现的问题；未来如果多模态大模型更完善，也许可以让 AI 直接看到运行结果的截图或视频，从而令这个过程更自动化；
4. 里程碑功能完成后，我一般会要求 coding agent 进行收尾处理：
   - 对做出修改的模块以及关联部分进行重新审视，确保代码风格和设计一致性，清理冗余代码和注释；如果出现了特别大的模块或者比较多重复代码，就要求它进行重构优化；
   - 更新 AGENTS.md 和 README.md 文档，记录关键决策和实现细节，更新项目状态等。

> 在划分里程碑时，我一般遵循尽可能小而独立且可验证的原则，也就是小步快跑，这样可以让 coding agent 更容易理解和实现每个里程碑的目标，同时也便于我进行测试和验证。

下面是到目前为止项目经历的主要里程碑：

1. **基本 GUI**：选定 [Flet](https://flet.dev/) 作为 GUI 框架，实现了基本的棋盘显示和交互，包括棋子绘制、用户点击响应等，也包含了与引擎通信的协议雏形，以及一个随机乱下的 mock 引擎，确保 GUI 能够与引擎进行基本通信；根据经验，我还让 AI 实现了一个命令行入口 `main.py` 来启动程序的各项功能，也方便以后添加各种参数进行测试；
2. **协议**：前一里程碑里缺乏棋局规则的实现，所以继续完善了协议和引擎的核心模块，实现 Reversi 的基本规则（合法着法判断、棋局结束条件等），实现了一个能按规则随机走棋的引擎；同步完善 GUI 与引擎之间的通信协议并单独维护了一个文档来记录此协议；
3. **GUI 完善**：不急于实现 AI 引擎，和随机落子的对手下几盘试试，继续改进发现的交互和界面问题，包括：增加了悔棋按钮、更美观的棋盘和棋子显示、窗口大小改变时自适应调整棋盘大小、自动高亮提示可落子的格子；
4. **内置引擎**：实现了一个基于 Minimax 加 Alpha-Beta 剪枝的 AI 引擎，可以作为人类玩家的对手；
5. **GUI 完善**：有了正儿八经的 AI 对手之后我就认真下了几盘，发现并解决了新需求：在无子可下时需要一个 Pass 按钮，以及需要一个实时显示双方得分的 scoreboard；
6. **重构**：下一步按原定计划要包装第三方库 [rust_reversi](https://github.com/neodymium6/rust_reversi) 中的几个引擎，在此之前按照我的要求进行了一组重构，把可复用的引擎代码移到 `base_engine` 模块中，把本地引擎 `local_engine` 改名 `minimax_engine` 等，coding agent 没有主动进行这些重构有点令人失望，让它改名它能把所有文档、代码都改对是不错的；
7. **包装外部引擎**：实现第三方引擎包装的过程是比较令我惊喜的，Copilot 和大语言模型配合完成了对 `rust_reversi` 库的接口探查，正确理解其接口中的各项概念并与我们设计的引擎及通信协议进行了对照，实现了两侧对棋盘、棋局规则和着法搜索概念上的兼容性包装，几乎一次成功，这些工作人类去做恐怕要花上十倍的时间；
8. **棋局保存与回放**：经过讨论最后选定 JSON 作为棋局保存文件的格式以提供最佳的人类可读性与平台中立性，然后依次实现了棋局回放、棋局持久化及 Save/Load 功能，中间也顺便对界面布局进行了优化，总的来说涉及到 GUI 部分比较复杂一点的要求都会有点磕磕碰碰；
9. **重构**：到这里包含 GUI 定义与处理的代码 `app.py` 已经非常庞大，于是在我的要求下对其进行了拆分重构，产生了分功能的多个模块，尽量降低主模块的复杂度；
10. **引擎对战**：现在开始可以选择引擎间对战了，这里是先实现 GUI 的引擎对战还是先实现命令行 headless 的引擎对战（甚至批量对战）呢？这次 AI 仍然倾向于先实现命令行和批量版本，这次我依了它，这部分实现非常顺利，但我发现两个引擎之间对战结果是高度重复的，每一局都严格一样，看来是引擎算法实现中缺乏随机性导致的；
11. **引擎强化**：对 `minimax` 引擎进行强化，引入了 `top_k` 和 `temperature` 等随机性控制参数，增加着法选择的随机性，另外把 `rust_reversi` 中另外两个引擎也包装导入进来了，算法大乱斗，启动；
12. **GUI 完善**：在 GUI 中对双方单独选择人类或引擎，并对引擎参数提供配置对话框，这部分涉及对历史代码的重构，以及对文档的更新同步；
13. **引擎辅助分析**：人类选手可以选择一个引擎提供辅助分析，在可选着法上显示价值评估，这个功能的实现一波三折，主要挑战是异步进程的管理，具体后面花絮会说；这也是到目前为止最后一个功能性里程碑；
14. **重构**：清理无用和不合理配置的代码（比如把应该在引擎处理的代码移到引擎侧），更新文档。

可以在项目的[提交历史](https://github.com/neolee/reversi/commits/main/)中看到这些关键里程碑。

> 我在自己写程序时倾向于频繁的提交，并在状态稳定之后通过 `rebase` 来整理提交历史；但在和 AI 合作时，我更倾向于让每个里程碑验证完成后对应一个提交，以便更清晰地记录项目的进展和决策过程。这也得益于 Copilot 提供了强大的 Undo/Redo 功能，我可以在里程碑内不必依赖于 `git` 来完成修改的回退。

## AI 助手花絮

在项目开发过程中，我的 AI 助手有过若干次栽了的时刻，让我们人类可以窥见 AI 的一些有趣的思维模式。

### GUI 难题

第一次比较大的挫折发生在实现棋盘自适应缩放功能时。一开始生成的代码棋盘大小是固定的，无论如何调整窗口大小，棋盘并不会跟着缩放。我让 AI 助手添加棋盘自适应缩放的功能，结果 Gemini 3 经过多轮尝试都无功而返，我把大语言模型切换到 GPT-5.1-Codex，但它也没辙。

我只好亲自去研究了相关代码，猜测可能是没有正确捕获窗口大小变化的事件，AI 写了很多代码实际上都没被运行，在代码中添加了必要日志输出后确认了这一点。在我把这个提示告知 AI 助手后，它终于实现了正确的功能。因为我完全不熟悉 Flet 这个框架，也没打算认真学习它，所以就没管了。

在很后期，我终于决定仔细审查一下 GUI 部分的代码、尤其是那些很冗长看上去很难解的代码时，发现 AI 助手实现的自适应缩放功能，竟然没使用 Flet 框架提供的窗口大小变化事件，而是自己创建了一个线程不断轮询窗口大小变化，我问它为啥，它回答了三点：
1. Flet 的窗口 `resized` 事件只在大小改变完成后触发，不能在用户拖曳鼠标改变窗口大小的过程中持续更新棋盘大小；
2. 它“觉得”轮询更简单直接，不需要理解 Flet 的事件机制；
3. 似乎预见到我对它的实现性能上的担忧，它主动解释说在轮询中增加了异步的 `await asyncio.sleep()` 调用，这样不会阻塞主线程。

客观地说，有它的道理，不过我还是毫不犹豫地叫它改用框架的标准事件。

AI 后面在处理 UI 布局时也遇到不少小挑战，比如我要求它实现引擎日志显示框的自适应布局，并让日志显示能随着内容增加自动滚动，这些对人类来说相对简单的操作，AI 反而花了更多的时间。

总的来说 AI 处理类似 Flet 这样并不是特别广泛使用的 GUI 框架时问题还是比较多的，一方面 GUI 应用运行实际效果需要人类转述给 AI，增加了理解偏差的环节，另一方面也感觉目前的 coding agent 对 GUI 代码的“架构良好”理解不是很够。后者我猜想可能与前端代码及相关文章的质量普遍就不是很高有一定关系。

### 领域洞察

在实现引擎辅助分析功能时，最开始的版本中分析引擎只会针对棋局现状给出一次评估，我提议如果人类棋手没落子，分析引擎应该自动加深搜索深度持续分析，即所谓的“迭代加深”搜索，当然有些算法不支持这样的迭代加深，但 Minimax 算法肯定是没问题的，AI 很快就实现了这个功能。

然后我发现 Minimax 算法提供的计算波动很大，比如一个格子一次是10分下一次是-10分再下次又是12分之类，这显然是因为评估局面轮到不同选手走棋时的现象，对此 AI 助手的理解也没问题，它向我解释这是 Minimax 算法的正常现象，并完全没打算做任何修改。我提醒它：这是不是意味着引擎在用于后台分析时最好自动选择偶数深度？就是一开始计算两步，然后再计算后两步，依次迭代加深，对此 AI 助手说：你的洞察很棒，确实如此，我们应该忽略用户配置的计算深度，而采用 `2,4,6,...` 这样递增的深度，然后就去改了。

所以我的感觉是，领域方面的洞察目前还得靠人。

### 异步难题

众所周知，多线程和多进程的异步程序开发即使对有经验的人类程序员来说也很棘手，那么对 AI 来说表现如何呢？

本项目开发过程中，直到最后一个功能里程碑之前都没遇到异步处理的问题，Python 内置的基于线程的异步机制透明又好用，但开发引擎辅助分析功能时，引擎分析需要在后台运行以免干扰 GUI 运行，而 `rust_reversi` 中的引擎调用外部 Rust 程序会导致 Python 的全局锁（GIL），所以这部分引擎必须运行在独立进程中，这一功能直接把本项目的难度上升了一档。

一开始 AI 并没有做多进程处理，我测试时发现使用 `rust_reversi` 的 `Minimax` 引擎做辅助分析会卡住 GUI（`rust_reversi` 的另两个引擎不会做迭代加深搜索，所以这个问题不明显），告诉 AI 助手之后它立刻意识到问题所在，并添加了进程管理的模块。

真正的问题出现在 GUI 程序退出时，当后台引擎进程正在进行分析时，如果 GUI 退出，后台进程仍会尝试与 GUI 进程进行通信，导致程序退出前报一个 `RuntimeError`。这对于多进程程序来说算常见问题，解决方案也不复杂，主进程应该在退出前确保其他进程的 *graceful shutdown*。但 AI 并没有很好的解决这个问题，它尝试在主进程退出前调用引擎对象的终止方法，但这其实不可靠，因为后台进程正忙着计算，可能无法立刻响应，最后还是我提示 AI 增加一个退出等待的对话框，并通过轮询等待后台进程正确结束再退出。

顺便，这次试用 [Flet](https://flet.dev/) 框架也发现一些问题。

比如在实现这个进程善后处理时，就发现 Flet 框架其实没提供应用程序退出事件的拦截机制，关闭主窗口倒是有个事件可以拦截，先处理完再继续关闭，但如果用户用快捷键直接退出应用（比如 Windows 下的 ALT-F4 或者 macOS 下的 Command-q）则完全无法拦截。

另一个问题是 Flet 的 `0.28.3` 版本有个 bug，导致 `FilePicker` 在 macOS 最新版本下无法显示文件选择对话框（与文件访问权限有关），回滚到 `0.28.2` 版本可以解决，或者等后续版本修正。

遇到这类问题，coding agent 如果不进行有效的全网搜索是很难做出正确判断的，通常需要人类干预。

### 小小年纪染上路径依赖

后期我在做代码 review 时发现 GUI 部分有大量代码用诸如 `page = getattr(self, "page", None)` 这样的写法来获取对象属性，我觉得很奇怪，就问它为啥不用更直接的 `self.page` 来访问属性，它回答说有些场景下该属性对象可能已经被销毁，这么写更安全，同时也可以防止 `pylance` 的类型警告。我马上指出有些场景明显 `self.page` 一定存在，它就说“有些时候绝对是有危险的，但如果你希望，我可以检查一遍把不需要这么写的地方改掉”，像极了某些人类程序员。在我的威逼下它检查了一圈最后把所有这样的写法都改了。

我猜测这是在之前代码生成过程中为了解决某些运行错误或者类型警告而引入的，然后因为目前 coding agent 普遍倾向于写与现有代码同型的代码，导致它在后续代码生成中不断重复这种写法，形成了一种路径依赖。

所以 AI 虽然年纪尚轻，但却显得老气横秋，对自己做过的事颇为顽固。可能是学习了太多人类的代码，心理年龄爆表了吧（大雾）。
